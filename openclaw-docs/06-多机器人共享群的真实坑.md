## 1. 共享群的复杂度为什么这么高

因为你同时在处理三层问题：

```
┌─────────────────────────────────────────────────────────────────┐
│                   共享群的三层问题                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   第 1 层：飞书消息分发                                         │
│   ───────────────────                                           │
│   消息是怎么投递的？谁能收到？                                  │
│                                                                 │
│   第 2 层：OpenClaw 路由决策                                    │
│   ───────────────────                                           │
│   收到消息后，谁来回复？                                        │
│                                                                 │
│   第 3 层：人设和群规则                                         │
│   ───────────────────                                           │
│   回复内容是否合适？角色边界清晰吗？                            │
│                                                                 │
│   ⚠️ 只要有一层不清楚，共享群就容易出现怪行为                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 典型问题场景

### 2.1 问题一：机器人总在历史话题下面回复

**表象：**
- 你明明发了一条新消息
- 机器人却总挂在很旧的那条消息下面回

**根因：**
- `replyInThread` 策略没配好
- 机器人沿着旧 thread 继续回复

**解决方案：**
```json
{
  "replyInThread": "disabled"
}
```

### 2.2 问题二：@ceo 结果 trade 也出来说话

**表象：**
- 用户明确 `@ceo`
- 结果 `trade` 也跳出来回复

**根因：**
- 没有配置点名优先逻辑
- 所有 bot 都认为自己应该回复

**解决方案：**
```json
{
  "requireMention": true,
  "mentionPriority": "strict"
}
```

### 2.3 问题三：bot @ 另一个 bot，对方不接

**表象：**
- 群里显示 `@trade`
- 但 `trade` 不一定会收到可处理的事件

**根因：**
- 飞书事件传递机制限制
- bot-to-bot @ 不可靠

**解决方案：**
- 不要依赖群里的 bot-to-bot @
- 使用内部 handoff 机制

## 3. 共享群配置详解

### 3.1 群组配置结构

```json
{
  "groups": {
    "oc_xxx_shared_group": {
      "replyToMode": "first",
      "replyInThread": "disabled",
      "requireMention": false,
      "systemPrompt": "群聊参与规则..."
    }
  }
}
```

### 3.2 配置字段说明

| 字段 | 可选值 | 说明 | 推荐值 |
|------|--------|------|--------|
| `replyToMode` | `first`, `none`, `always` | 回复时是否引用原消息 | `first` |
| `replyInThread` | `enabled`, `disabled` | 是否沿着 thread 回复 | `disabled` |
| `requireMention` | `true`, `false` | 是否必须 @ 才回复 | `false`（调试）|
| `systemPrompt` | 字符串 | 群聊专用提示词 | 见下文 |

### 3.3 字段详解

#### replyToMode

```
┌─────────────────────────────────────────────────────────────────┐
│                   replyToMode 选项                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   "first"  - 引用原消息一次，不重复引用                         │
│              ┌────────────────┐                                 │
│              │ 你好           │                                 │
│              ├────────────────┤                                 │
│              │ > 你好         │                                 │
│              │ 你好！有什么... │                                 │
│              └────────────────┘                                 │
│                                                                 │
│   "none"   - 不引用，直接回复                                   │
│              ┌────────────────┐                                 │
│              │ 你好           │                                 │
│              ├────────────────┤                                 │
│              │ 你好！有什么... │ ← 无引用                       │
│              └────────────────┘                                 │
│                                                                 │
│   "always" - 每段都引用（容易刷屏）                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**推荐**：`first` - 既保留上下文，又不会刷屏

#### replyInThread

```
┌─────────────────────────────────────────────────────────────────┐
│                   replyInThread 选项                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   "enabled"  - 沿着旧话题链继续回                               │
│                问题：容易钻进历史回复链                          │
│                                                                 │
│   "disabled" - 回复放回当前对话上下文                           │
│                推荐：避免话题混乱                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**推荐**：`disabled`

## 4. 共享群 System Prompt 模板

### 4.1 推荐模板

```markdown
# 群聊参与规则

你是一个 AI 团队成员，和其他 AI 同事一起在这个群里协作。

## 回复原则

像人类成员一样先观察。只有在以下情况才回复：

1. **被明确点名/提问** - 消息中包含 @你的名字
2. **你能提供实质性纠错或增量信息** - 发现明显错误或能补充重要信息
3. **能推动决策** - 你的专业领域能帮助解决问题

## 不回复的情况

- 只是闲聊或打招呼
- 重复其他人已经说过的内容
- 没有增量信息
- 不需要你介入的领域

## 回复格式

如果判断不需要回复，严格只输出：NO_REPLY

如果需要回复，直接输出回复内容，不要解释为什么回复。
```

### 4.2 配置示例

```json
{
  "groups": {
    "oc_xxx_shared_group": {
      "replyToMode": "first",
      "replyInThread": "disabled",
      "systemPrompt": "群聊参与规则：像人类成员一样先观察。只有在以下情况才回复：被明确点名/提问、你能提供实质性纠错或增量信息、或能推动决策。若只是闲聊、重复、无增量信息、或不需要你介入，请严格只输出 NO_REPLY。"
    }
  }
}
```

## 5. 点名优先逻辑

### 5.1 逻辑说明

```
┌─────────────────────────────────────────────────────────────────┐
│                   点名优先逻辑                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   消息内容                      谁可以回复                      │
│   ─────────────────────────────────────────────────────────────│
│                                                                 │
│   "@ceo 你来处理"               只有 ceo                        │
│                                  其他 bot 跳过                   │
│                                                                 │
│   "@ceo @trade 你们一起看"      ceo 和 trade 都可进入判断       │
│                                  其他 bot 跳过                   │
│                                                                 │
│   "大家好"（没 @ 任何 bot）     所有 bot 都可进入判断           │
│                                  由 prompt 决定是否回复          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 重要边界

> ⚠️ 修了"抢答"不等于封死未点名消息

真实逻辑是：
- 如果消息明确 `@` 了某个已配置 bot
- 且当前 bot 不在被点名名单里
- 当前 bot 才直接跳过

**不是**"只要没 @ 自己，机器人一律不许回"。

## 6. Bot-to-Bot 协作的正确方式

### 6.1 不要依赖群里互相 @

```
❌ 不推荐的方式：
───────────────────
用户: @ceo 帮我查一下代码
CEO: @trade 你来处理
Trade: 好的，我来...（可能收不到事件）
```

问题：飞书事件传递不稳定，bot 可能收不到 @ 消息。

### 6.2 使用内部 Handoff

```
✅ 推荐的方式：
───────────────────
用户: @ceo 帮我查一下代码
CEO: （内部调用 trade）好的，我来帮你查...

用户只看到 CEO 在回复，但实际是 trade 在处理
```

### 6.3 配置 agentToAgent

```json
{
  "tools": {
    "agentToAgent": {
      "allow": true
    }
  }
}
```

## ⚠️ 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 所有 bot 都回复同一条消息 | 没有配置点名优先 | 设置 `requireMention` 或添加 system prompt |
| 回复总挂在旧消息下面 | `replyInThread` 为 enabled | 设置为 `disabled` |
| bot 之间协作失败 | 依赖群里 @ | 使用内部 handoff |
| @指定 bot 但其他也回 | 点名逻辑未生效 | 检查配置是否正确应用 |
| 回复刷屏 | `replyToMode` 为 always | 改为 `first` |

## 8. 共享群的真正目标

我后来对共享群的理解变成了：

> **共享群不是"让所有 bot 都抢着发言的地方"，而是"让多个 bot 在同一个场域下有秩序地存在"的地方。**

如果你的配置做到了下面这几点，共享群基本就算稳定了：

1. ✅ 用户点名谁，谁优先处理
2. ✅ 非目标 bot 不乱插嘴
3. ✅ 没有 thread 混乱
4. ✅ 没有刷屏
5. ✅ 真有增量信息时，系统仍然允许补充
