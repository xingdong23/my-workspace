## 3. 排障命令速查

### 3.1 状态检查

```bash
# 查看整体状态
openclaw status

# 环境诊断
openclaw doctor

# 配置验证
openclaw config validate

# 网关状态
openclaw gateway status
```

### 3.2 服务管理

```bash
# 重启网关
openclaw gateway restart

# 停止网关
openclaw gateway stop

# 启动网关
openclaw gateway start
```

### 3.3 日志查看

```bash
# 查看实时日志
openclaw logs --follow

# 查看最近 100 行
openclaw logs --lines 100

# 只看错误
openclaw logs --level error

# 查看特定 agent 日志
openclaw logs --agent ceo
```

### 3.4 插件检查

```bash
# 列出已安装插件
openclaw plugins list

# 查看插件状态
openclaw plugins status feishu
```

## 4. 日志和 Transcript 位置

### 4.1 关键文件位置

| 文件/目录 | 位置 | 内容 |
|----------|------|------|
| 主配置 | `~/.openclaw/openclaw.json` | 全局配置 |
| Gateway 日志 | `~/.openclaw/logs/gateway.log` | 网关运行日志 |
| Agent 日志 | `~/.openclaw/agents/{id}/agent/logs/` | Agent 运行日志 |
| Session | `~/.openclaw/agents/{id}/agent/sessions/` | 会话数据 |
| Transcript | Session 目录内的 `.json` 文件 | 完整对话记录 |

### 4.2 源码位置（Homebrew 安装）

```
/opt/homebrew/
├── bin/openclaw                              # 命令入口
└── lib/node_modules/openclaw/                # 包目录
    ├── dist/                                 # 编译后代码
    └── extensions/                           # 插件目录
        └── feishu/
            └── src/                          # 飞书插件源码
```

## 5. 问题层级判断

### 5.1 配置层问题

**典型表现：**
- `openclaw config validate` 报错
- binding 不生效
- account 没对上
- agent 根本没被路由到

**排查方法：**
```bash
openclaw config validate
openclaw status
```

**解决方向：** 修改 `openclaw.json`

### 5.2 Prompt 层问题

**典型表现：**
- 系统已经拿到了正确输入
- 但回复风格、角色边界、语气不符合预期

**排查方法：**
- 查看 transcript 确认输入正确
- 检查 `IDENTITY.md`、`SOUL.md`

**解决方向：** 优化 workspace 文件

### 5.3 插件层问题

**典型表现：**
- 输入在进入模型前就已经变形了
- 某些字段丢了
- mention/thread 信息在入站时没保留

**排查方法：**
```bash
# 查看 gateway 日志
openclaw logs --follow

# 查看 session transcript
cat ~/.openclaw/agents/ceo/agent/sessions/xxx/transcript.json
```

**解决方向：** 修改插件源码

## 6. 源码热修步骤

### 6.1 确认问题层级

在修改源码之前，确认：
1. ✅ 配置层没问题（`config validate` 通过）
2. ✅ 状态正常（`status` 显示 running）
3. ✅ 日志/转录显示问题在插件层

### 6.2 定位源码位置

```bash
# 找到 OpenClaw 安装位置
which openclaw
# 输出: /opt/homebrew/bin/openclaw

# 查看实际包目录
ls -la /opt/homebrew/bin/openclaw
# 输出: -> ../lib/node_modules/openclaw/bin/openclaw

# 进入插件目录
cd /opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src
```

### 6.3 修改源码

**原则：**
- 只改最小闭环
- 保留原代码注释
- 记录修改原因

**示例修改（消息解析）：**

```typescript
// 原代码
const text = message.content;

// 修改后
// FIX: 保留 mention 信息，只去掉当前 bot 的 mention
const text = preserveMentions(message.content, currentBotId);
```

### 6.4 重启生效

```bash
# 重启网关使修改生效
openclaw gateway restart

# 验证修改
openclaw logs --follow
```

## 7. 常见问题修复示例

### 7.1 抢答问题

**问题：** `@ceo` 时 `trade` 也回复

**根因：** 飞书插件解析消息时丢失了 mention 目标

**修复思路：**
```
┌─────────────────────────────────────────────────────────────────┐
│                   抢答问题修复思路                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 保留显式 mention 目标                                      │
│      └── 解析消息时提取所有 @ 目标                              │
│                                                                 │
│   2. 只去掉"当前 bot 自己"的 mention                            │
│      └── 其他 bot 的 mention 保留                               │
│                                                                 │
│   3. 路由判断                                                   │
│      └── 如果消息明确 @ 了别的 bot，当前 bot 跳过               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 Thread 问题

**问题：** 机器人总在旧消息下面回复

**根因：** `replyInThread` 逻辑错误

**修复思路：**
- 检查 `replyInThread` 配置
- 必要时修改插件的回复逻辑

## 8. 源码热修注意事项

### 8.1 必须遵守

```
┌─────────────────────────────────────────────────────────────────┐
│                   源码热修规范                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ✅ 必须做：                                                   │
│   • 先拿到日志和 transcript 证据                                │
│   • 明确是哪一层有问题                                          │
│   • 只改最小闭环                                                │
│   • 改完立刻回归测试                                            │
│   • 记录补丁内容                                                │
│                                                                 │
│   ❌ 不要做：                                                   │
│   • 没证据就大改插件                                            │
│   • 同时改多处代码                                              │
│   • 改完不测试就上线                                            │
│   • 不记录修改内容                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 升级会覆盖补丁

> ⚠️ `openclaw update` 很可能覆盖你改过的内容

**建议：**
- 短期：直接 patch 安装目录，先解决问题
- 长期：保留补丁记录，或做成本地覆盖插件

## ⚠️ 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 改了源码不生效 | 没重启网关 | `openclaw gateway restart` |
| 找不到源码位置 | 安装方式不同 | `which openclaw` 查找 |
| 升级后问题复现 | 补丁被覆盖 | 重新应用补丁或做成本地插件 |
| 日志看不到详情 | 日志级别不够 | `openclaw logs --level debug` |
| transcript 找不到 | session 被清理 | 重新触发问题，立即查看 |

## 9. 高级用户的系统感

到了这一步，真正重要的不是你会不会改一段代码，而是你有没有形成这种系统感：

```
┌─────────────────────────────────────────────────────────────────┐
│                   系统感核心问题                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   • 问题是从哪里进入的？                                        │
│   • 在哪一层被处理？                                            │
│   • 在哪一层信息丢失了？                                        │
```
