## 1. 一个核心认知

**市面上的各种 OpenClaw，本质上都是同一套内核的不同部署形态。**

真正的差异不在"是不是 OpenClaw"，而在：

| 差异点 | 选项 |
|--------|------|
| Gateway 跑在哪里 | 本机 / App托管 / VPS / Docker |
| 状态文件放哪 | 本地磁盘 / 容器volume / 远程主机 |
| 谁来管理生命周期 | 你自己 / App / Docker |
| 安全边界怎么画 | 本地loopback / 公网暴露 |

## 2. 五种部署方式对比

### 方式一：本机 CLI（推荐新手）

```
你的电脑
├── OpenClaw CLI
├── Gateway（你手动启动）
├── Dashboard（本地访问）
└── 所有配置和日志
```

**优点：**
- 最容易理解和排障
- 路径、日志都在自己机器上
- 官方文档完全对应

**缺点：**
- 电脑关机 = bot 下线
- 依赖本机环境

**适合：** 第一次接触、想先跑明白的人

---

### 方式二：官方 App 托管

```
你的电脑
├── OpenClaw.app
│   └── 自动管理 Gateway
├── Dashboard（App 内置）
└── 配置文件
```

**优点：**
- 更像产品，体验友好
- 生命周期自动管理
- 少动命令行

**缺点：**
- 透明度不如 CLI
- 深度排障还是要回命令行

**适合：** macOS 用户、想省心的人

---

### 方式三：远程 VPS

```
你的电脑（控制端）
    │
    └── SSH/网络 ──▶ VPS（24小时在线）
                        ├── Gateway
                        ├── 所有配置
                        └── Dashboard（远程访问）
```

**优点：**
- 24/7 在线
- 不依赖你电脑开机
- 适合长期挂机器人

**缺点：**
- 运维复杂度高
- 需要处理主机安全
- 问题排查要"本地+远端"双环境

**适合：** 需要 24 小时在线、有 Linux 运维基础的人

---

### 方式四：Docker

```
Docker 容器
├── Gateway
├── 配置（volume 挂载）
└── 日志（volume 挂载）
```

**优点：**
- 环境标准化
- 迁移、回滚方便
- 不污染宿主机

**缺点：**
- 多了一层抽象
- 排障更复杂

**适合：** 有 Docker 经验、想标准化环境的人

---

### 方式五：社区一键部署（Coolify 等）

```
Coolify / Docker Compose
├── OpenClaw
├── 反向代理
├── 额外工具
└── 预配置环境
```

**优点：**
- 开箱即用
- 集成了周边工具

**缺点：**
- 引入更多中间层
- 出问题难定位
- 可能混入不明代码

**适合：** 有容器经验、能独立排查问题的人

> ⚠️ **建议**：先理解官方原版，再上第三方封装

## 4. 安全边界

### 4.1 核心风险

OpenClaw 这类系统的风险不是"说错话"，而是：

| 风险 | 说明 |
|------|------|
| **文件访问** | agent 能访问你的哪些文件 |
| **工具调用** | agent 能执行什么命令 |
| **网络暴露** | Gateway 是否暴露在公网 |
| **密钥泄露** | API key、appSecret 是否安全 |

### 4.2 权限设置

```bash
# 收紧主目录权限
chmod 700 ~/.openclaw

# 收紧配置文件权限
chmod 600 ~/.openclaw/openclaw.json

# 敏感文件
chmod 600 ~/.openclaw/agents/*/agent/auth-profiles.json
chmod 600 ~/.openclaw/agents/*/sessions/sessions.json
```

### 4.3 网络安全

| 方式 | 安全性 | 说明 |
|------|--------|------|
| 本地 loopback | ⭐⭐⭐⭐⭐ | 最安全，只能本机访问 |
| SSH tunnel | ⭐⭐⭐⭐ | 推荐，加密隧道 |
| Tailscale | ⭐⭐⭐⭐ | 推荐，私有网络 |
| 公网暴露 | ⭐ | 危险，不推荐 |

### 4.4 密钥管理

```
┌─────────────────────────────────────────────────────────────────┐
│                   密钥安全原则                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ✅ 必须做：                                                   │
│   • 密钥泄露后立即轮换                                          │
│   • 不要在截图、日志中暴露密钥                                  │
│   • 不同环境用不同密钥                                          │
│                                                                 │
│   ❌ 不要做：                                                   │
│   • 把密钥提交到 git                                            │
│   • 在群里发配置截图不遮挡                                      │
│   • 所有服务用同一个密钥                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 5. 成长路径建议

### 阶段一：跑起来
- 本机 CLI + 本机 Gateway
- 1-2 个 agent
- 目标：理解系统

### 阶段二：跑稳定
- 多 agent + skills
- 共享群策略
- 目标：日常可用

### 阶段三：跑专业
- 远程部署 / Docker
- 源码定制
- 目标：深度掌控
