## 1. 十大踩坑总结

### 坑 1：能运行 ≠ 配置已收口

**现象：** 系统能跑，但角色混乱，文档、配置、运行目录不一致

**根因：** 活配置和历史配置混在一起

**解决：**
- `openclaw.json` 成为唯一真源
- 当前活跃 agent 只有一套定义
- workspace、binding、accountId、运行目录语义一致

---

### 坑 2：历史遗留会持续污染系统

**现象：** 路由混乱、dashboard 噪音大、排障困难

**根因：** 旧目录、旧 session、旧通道没清理

**解决：**
- 活配置只保留当前还在用的
- 旧目录归档或删除
- 不再使用的 channel 直接删除

---

### 坑 3：API 测试能通 ≠ Agent 能跑

**现象：** `curl` 测试通过，但真实 agent 请求失败

**根因：** OpenClaw 真实请求会叠加大量上下文

**解决：**
- 用真实 agent 请求测试
- 不要只测最小 API 请求

---

### 坑 4：模型问题可能是协议路径问题

**现象：** 模型名对、key 有效，但 OpenClaw 不通

**根因：** 走错了 provider 适配层

**解决：**
- 分清模型名、协议类型、实际 baseUrl
- 检查是否用了正确的 coding plan 路径

---

### 坑 5：共享群怪行为 ≠ 模型幻觉

**现象：** `@ceo` 时 `trade` 也回复

**根因：** 飞书插件解析时丢失了 mention 信息

**解决：**
- 先看 transcript 里 agent 实际看到了什么
- 如果 mention 信息没了，改插件而不是改 prompt

---

### 坑 6：bot-to-bot 群接力不可靠

**现象：** bot 在群里 @ 另一个 bot，对方不响应

**根因：** 飞书事件传递机制限制

**解决：**
- 使用内部 handoff 机制
- 不要依赖群里的 bot-to-bot @

---

### 坑 7：replyToMode 和 replyInThread 影响体验

**现象：** 机器人总挂到历史话题下面回复

**根因：** 消息发送策略配置错误

**解决：**
- `replyToMode: "first"`
- `replyInThread: "disabled"`

---

### 坑 8：Dashboard 异常可能只是前端问题

**现象：** skills 页面为空、会话页堆满过期会话

**根因：** 筛选条件错误或索引没清

**解决：**
- 用 CLI 命令确认真实状态
- 不要先被 UI 吓到

---

### 坑 9：热修有效但升级会覆盖

**现象：** 改了源码有效，但升级后问题复现

**根因：** `openclaw update` 覆盖补丁

**解决：**
- 短期：安装目录热修
- 长期：保留补丁记录或做成本地插件

---

### 坑 10：角色模糊 + 能力泛滥 = 混乱

**现象：** 多个 bot 都想回答、skill 边界模糊

**根因：** 没有明确的角色边界

**解决：**
- 身份更清楚
- 路由更明确
- skills 更聚焦
- 共享群规则更收紧

## 2. 问题层级速查表

| 现象 | 可能层级 | 首先检查 |
|------|---------|---------|
| 配置不生效 | 配置层 | `openclaw config validate` |
| agent 不响应 | 配置层 | `openclaw status` |
| 回复风格不对 | Prompt 层 | `IDENTITY.md`、`SOUL.md` |
| mention 丢失 | 插件层 | transcript + 日志 |
| 群里抢答 | 插件层/配置层 | transcript + groups 配置 |
| Thread 混乱 | 配置层 | `replyInThread` |
| API 报错 | Provider 层 | 协议路径、key 有效性 |

## 5. 最后的判断

**OpenClaw 真正难的，不是写配置，而是形成系统观。**

也就是你能不能看清：

- 问题发生在配置层、运行层、通道层，还是插件层
- 哪些现象只是表象
- 哪些才是真正的技术根因

一旦你形成了这种判断方式，OpenClaw 就不再是一个难以捉摸的黑盒。

**这时候你不只是"会用 OpenClaw"，而是开始真正"掌控 OpenClaw"。**
