# 06.5 Bot-to-Bot Relay：为什么飞书群里不通，以及如何改

## 问题是什么

很多人第一次做多机器人共享群时，直觉会是这样：

1. 人类在群里 `@ceo`
2. `ceo` 在群里再 `@trade`
3. `trade` 接着以自己的身份继续回复

但在实际运行里，这条链路通常不可靠。现象是：

- 群里肉眼能看到 bot A `@` 了 bot B
- 人类成员也都能看到这条消息
- 但 bot B 后台没有继续接话

这不是单纯 prompt 不够聪明，也不是 `groupPolicy` 配错了。

## 结论

**根因在飞书事件模型，不在 OpenClaw 的群策略。**

更准确地说：

- 飞书会把**用户发送的群消息**推给机器人
- 但通常不会把**机器人发送的群消息**再作为另一个机器人的标准入站事件推过去
- 所以 bot A 在群里 `@` bot B，UI 层可见，不等于 bot B 后台真的收到了一条可处理的消息

因此，飞书群里的 bot-to-bot `@`，**不应该被当成可靠协作机制**。

---

## 为什么这么判断

### 1. 飞书官方文档已经给出了核心限制

飞书官方“接收消息”文档写得很清楚：

- 这是“**机器人接收到用户发送的消息后**”触发的事件
- 如果应用具备 `im:message.group_msg` 权限，可以接收群里**用户发送的所有消息**
- 文档还明确写了：**不包含机器人发送的消息**

官方文档：

- [飞书开放平台：接收消息](https://open.feishu.cn/document/server-docs/im-v1/message/events/receive)

同样，飞书官方“回复消息”文档的返回样例里，机器人自己发出去的消息 `sender_type` 是：

- `app`

官方文档：

- [飞书开放平台：回复消息](https://open.feishu.cn/document/server-docs/im-v1/message/reply)

这两条信息合起来，结论就很直接：

- **receive 事件针对的是用户消息**
- **机器人发出的消息属于 `sender_type=app`**
- 所以 bot A 发出的群消息，通常不在 bot B 的标准接收范围里

### 2. OpenClaw 官方也没有承诺内置 bot-to-bot bridge

OpenClaw 官方 FAQ 的原话非常关键：

> There is no built-in "bot-to-bot" bridge

官方文档：

- [OpenClaw FAQ](https://docs.openclaw.ai/help/faq)

这句话的意义不是“做不到”，而是：

- OpenClaw 本身没有帮你把“bot A 的出站消息”自动转成“bot B 的入站消息”
- 如果要做这件事，需要你自己补一层 bridge / relay

### 3. 本机 OpenClaw 的 Feishu 插件其实已经尽力了

本机这套 Feishu 插件已经做了几件正确的事：

- 能解析普通消息里的 `mentions`
- 能解析富文本 `post` 里的 `<at>`
- 能识别人类到底 `@` 了哪个已配置 bot
- 未被点名的 bot 会主动跳过

相关代码位置：

- `bot.ts` 里对 mention 的识别和提取  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/bot.ts`
- `post.ts` 里对富文本 `<at>` 的解析  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/post.ts`
- `mention.ts` 里对 mention target 的提取  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/mention.ts`

也就是说：

- **人类 `@ceo` / `@trade` 这条链路现在是正常的**
- 问题不在“OpenClaw 看不懂人类点名了谁”
- 问题在“飞书不会把 bot 发的群消息再回推给另一个 bot”

### 2. 人类消息的路由是正常的

本机日志已经能证明：

- 同一条人类群消息会被多个 bot 看到
- 未被点名的 bot 会因为“明确点到了别的已配置 bot”而跳过

例如人类 `@ceo` 时，日志会出现：

- `trade` 收到消息，但因为点名的是 `ceo`，所以跳过

- `/Users/xingdong/.openclaw/logs/gateway.log`

### 3. 但没有看到“bot 发送的群消息再次变成另一个 bot 入站事件”的证据

这次调研里，最关键的现象不是“界面上看到了什么”，而是：

- 日志里能稳定看到人类消息进入
- 但没有证据表明“bot A 发到群里的消息”再次以标准 `receive_v1` 事件进入 bot B

这和飞书官方文档的事件模型是对得上的。

---

## 所以到底哪里不行

不是下面这些：

- 不是 `groupPolicy`
- 不是 `requireMention`
- 不是 `replyToMode`
- 不是模型理解力不够
- 不是 OpenClaw 完全不支持多 bot

真正不行的是这一步：

**“bot A 在飞书群里发出去的消息，会不会自动变成 bot B 的标准入站消息？”**

答案通常是：

**不会。**

你可以把它理解成两层：

- **展示层**：群成员都能看见 bot 消息
- **事件层**：bot B 后台不一定收到可处理事件

UI 可见，不等于 webhook / WebSocket 入站存在。

---

## 配置能不能解决

**不能。**

这件事不是再改几条 `openclaw.json` 就能变出来的。

原因很简单：

- 配置只能影响“消息已经进入 OpenClaw 之后怎么处理”
- 但这里的问题发生在更前面：**飞书根本没把 bot 消息作为 bot B 的入站事件推过来**

所以：

- 配置层无解
- prompt 层无解
- 真要做，只能补 relay / bridge

---

## 最可行的改法：自定义 bot-to-bot relay

### 思路

不要等飞书回推 bot A 的消息。

而是在 OpenClaw Feishu 插件内部主动做这件事：

1. bot A 生成最终回复
2. 插件识别这条回复里显式 `@` 了某个已配置 bot
3. 不等飞书平台回推
4. 直接在 OpenClaw 内部构造一条 synthetic inbound message
5. 把这条 synthetic message 交给目标 bot
6. 目标 bot 再以自己的 Feishu account 向同一个群发言

也就是说，不走“平台回推”，而走“插件内部中继”。

### 这套设计的优点

- 不依赖飞书是否回推 bot 消息
- 可以保持“人类看起来像 bot A 交接给 bot B”
- 仍然由目标 bot 自己发声，身份清晰

### 这套设计的难点

难点不在“让 B 说话”，而在“别失控”：

- bot A `@trade`，trade 回复里又 `@ceo`，容易循环
- 多账号环境下，可能重复 relay
- 同一条 relay 重试后可能重复发送
- 一个消息同时 `@trade @writer` 时，是否允许多播
- 目标 bot 的回复要挂在哪条消息下

---

## 建议的实现方式

### 版本 1：最小可用版

先做最小闭环，不做复杂编排：

- 只支持“显式 `@` 一个已配置 bot”
- 只支持 relay 一跳
- relay 只在指定群启用
- synthetic message 禁止再次 relay

这版的目标不是“完美”，而是“稳定可用”。

### 关键保护规则

如果做 relay，至少要加这些保护：

1. **只在显式 mention 已配置 bot 时触发**
2. **一次只允许 relay 给一个目标 bot**
3. **`maxHops = 1` 或 `2`**
4. **synthetic relay 消息不允许再次 relay**
5. **用 `originMessageId + targetAgentId` 做持久化去重**
6. **只在白名单群启用**

没有这些保护，最后很容易变成 bot 在群里自己对自己讲话。

### 最适合下手的代码位置

如果基于你现在这套 OpenClaw 安装包改，最合适的下手点是：

- Feishu 入站消息构造：  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/bot.ts`
- Feishu 回复分发：  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/reply-dispatcher.ts`
- Feishu 实际发送：  
  `/opt/homebrew/lib/node_modules/openclaw/extensions/feishu/src/send.ts`

原因是：

- mention 解析、dispatch、send，这三段都已经现成
- 不需要改 OpenClaw 的核心协议层
- 主要是补“出站转下一站入站”的中继逻辑

## 开发难度评估

如果只做你自己这台机器的定制版，大致是：

- **POC**：半天到 1 天
- **能长期用的稳定版**：2 到 4 天
- **带配置项、测试、文档的完整版本**：4 到 7 天

所以它不是特别夸张的工程，但也不是“一小时小补丁”。

更准确地说：

- **技术上可做**
- **系统上要克制**

---

## 我给你的建议

如果你只是想让群里 bot 协作“看起来成立”，这件事值得做。  
但不要一上来就做成通用桥接层。

建议顺序是：

1. 先做指定群的 relay MVP
2. 只支持一跳 relay
3. 只支持显式 mention
4. 跑稳定后，再考虑多跳或多播

如果后面真的要上生产心态用，最好把它做成：

- 可开关
- 可按群配置
- 可按 agent 白名单配置

---

## 最后的结论

这件事之所以让人误判，是因为：

- 群里明明看到了 bot A `@` bot B
- 人会自然以为 bot B “也看到了”

但对机器人系统来说，真正重要的不是“看见 UI”，而是：

- **有没有收到正式入站事件**

飞书这层通常没有给你这条链路。  
所以 bot-to-bot 在飞书群里不可靠，不是错觉，也不是 OpenClaw 太笨，而是：

**平台事件模型 + OpenClaw 无内置 bridge 的共同结果。**

如果要做，就不要继续赌平台回推，直接做插件内 relay。
